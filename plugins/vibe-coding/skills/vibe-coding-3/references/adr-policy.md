# ADR Policy Reference

Use ADRs for durable decisions that shape the project.

## Record An ADR When

- Choosing B+ over pure skill or immediate plugin packaging.
- Changing hook policy or safety behavior.
- Changing verification fingerprint semantics.
- Changing scaffold structure.
- Introducing a new runtime dependency.
- Changing security posture.

## Do Not ADR

- Temporary todo items.
- Minor typo fixes.
- One-off command choices.
- Ordinary progress updates.

## Minimal ADR Shape

```markdown
# ADR-0001: Decision Title

**Date**: YYYY-MM-DD
**Status**: accepted
**Deciders**: project owner, Codex

## Context

Why this decision matters.

## Decision

What is now true.

## Alternatives Considered

What was rejected and why.

## Consequences

Positive, negative, and risk notes.
```
